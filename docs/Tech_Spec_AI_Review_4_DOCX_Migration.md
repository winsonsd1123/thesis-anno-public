# AI 审阅引擎架构升级：从 PDF 到 DOCX 全面迁移方案

## 0. 背景与动机
在 MVP 阶段使用大模型原生解析 PDF 遇到不可逾越的瓶颈：**多模态大模型无法精准提取和校验像素级物理排版（如字体、字号、行距等）**。国内高校对论文排版有极其严苛的硬性要求，纯依赖大模型会导致严重的幻觉。
同时，PDF 提取的纯文本存在换页截断、页眉页脚干扰等问题，严重影响逻辑和 AI 痕迹审查的质量。

为此，系统核心架构从“PDF 视觉直给”全面升级为“DOCX 结构化深度解析”。

---

## 1. 前端交互与计费系统改造

### 1.1 文件上传与前端展示
- **格式严格限制**：前端 `<input>` 控件的 `accept` 属性及文件校验逻辑从 `.pdf` 彻底改为 `.docx` (`application/vnd.openxmlformats-officedocument.wordprocessingml.document`)。**必须严格封杀旧版 `.doc` 格式**，因其为二进制格式，无法被本系统的 XML 解析器读取。
- **无预览架构**：彻底放弃前端对论文正文的直接预览（取消原 PDF iframe 预览），降低前端复杂度，UI 完全转向 DeepResearch 风格的“对话气泡 + 报告卡片”。

### 1.2 计费策略变更：从“按页”到“按字数”（服务端强制校验）
- **废弃 PDF 页数统计**：删除 `pdf-page-count.ts` 工具。
- **服务端字数统计（防篡改与防卡死）**：
  - 严禁在前端计算字数以防止浏览器主线程假死和恶意篡改计费参数。
  - **交互流程**：用户上传文件直传 Supabase Bucket -> 前端调用轻量级 Server Action (`POST /api/reviews/analyze-cost`) -> 服务端流式读取 DOCX 并提取纯文本计算精确字数 -> 返回需扣点数。
  - 前端展示：“检测到本文 3.2 万字，需扣除 2 点，是否开始审阅？” -> 用户确认后，真正创建 `review` 记录并触发 Trigger.dev。
- **阶梯扣点规则**：
  - `< 30,000 字`：扣除 1 个点数
  - `30,000 - 50,000 字`：扣除 2 个点数
  - `> 50,000 字`：扣除 3 个点数
- **数据库**：原 `reviews.pages` 字段概念转换为 `word_count`（可修改表结构或直接复用字段存储字数，前端展示文案对应修改）。

## 2. 核心底座：统一文档解析层 (Unified Parser Layer)

在 Trigger.dev 的后台任务队列中，在真正调用 LLM 前，必须经过**“统一解析层”**，这套 Hybrid Parser（混合解析器）是整个架构重构的灵魂。

### 2.1 Hybrid Parser 设计（双管齐下）
1. **内容主干（Mammoth.js）**：
   - 负责将 DOCX 转换为结构极其干净的 **Markdown 纯文本**。
   - 完美处理段落、列表、表格。这段 Markdown 是后续逻辑、AI 痕迹、参考文献审查的唯一文本输入源。
2. **样式侧枝（OpenXML AST + 文本锚点）**：
   - 负责精准获取像素级排版数据。
   - **解压内存刺客排雷**：严禁使用 `adm-zip` 全量读入内存，必须使用基于 Stream（流）的解压库（如 `yauzl`），边解压边处理，防止 Trigger.dev 发生 OOM。
   - 使用 `fast-xml-parser` 扫描 `document.xml` 和 `styles.xml`。**提取样式时，必须同时提取对应的纯文本片段**，生成长如 `{ text: "第一章 绪论", font: "宋体", size_pt: 16 }` 的 AST 节点。确保规则引擎发现排版错误时，能返回 `quote_text` 供用户精准定位。

### 2.2 防爆内存的图片处理策略 (Anti-OOM)
DOCX 内部的 `word/media/` 通常包含未压缩的原图。为防止 Trigger.dev 队列并发时导致内存溢出（OOM），实施以下硬性策略：
1. **流式拦截**：在 Mammoth 解析遇到图片时，不直接放入内存转换。
2. **Sharp 强制降采样**：使用 Node.js 的 C++ 底层库 `sharp`，将图片强制压缩并降采样（如限制最大宽度 1024px，转为高压缩比 WebP 或 JPEG），使 5MB 的图骤降至百 KB 级别。
3. **并发与熔断**：
   - 使用带并发限制的 `Promise.all` 处理图片。
   - **数量熔断**：设定阈值（例如单篇论文最多处理前 50 张图），超出部分直接丢弃，替换为 `[图片由于数量限制被忽略]` 文本锚点。
4. **多模态注入**：压缩后的图片转为 Base64，与 Markdown 文本组装成 Vercel AI SDK 支持的 `image` 类型多模态消息数组。

### 2.3 Trigger.dev 调度编排与数据复用流 (Data Flow)
**严禁各引擎重复解析！** 为了贯彻 DRY 原则并节省极大的 CPU/内存开销，文档解析必须在 Trigger.dev 的主编排流程中作为**全局前置节点**执行，且只执行一次。

**执行时序约束**：
1. **统一解析 (跑 1 次)**：主任务 `orchestrate-review` 下载 DOCX，调用 Unified Parser，在内存中生成唯一的数据副本 `{ markdownText, styleAST, images }`。
2. **文献预提取 (跑 1 次)**：利用代码（非 LLM）直接从 `markdownText` 中抠出参考文献数组 `referenceList`。
3. **并发下发 (传引用)**：将上述解析结果的内存引用，并发分发给四大引擎。各引擎只消费自己需要的数据部分：
   - `Logic`: 消费 `markdownText` + `images`
   - `Format`: 消费 `markdownText` + `styleAST`
   - `AITrace`: 消费 `markdownText`
   - `Reference`: 直接消费 `referenceList` (跳过全文处理)

## 3. 审阅引擎 (Review Engines) 适配改造

底层数据流变为 `{ markdownText, styleAST, imageDictionary }`，四大引擎全面重构：

### 3.1 逻辑审查引擎 (Logic Service)
- **输入变更**：接收清洗后的 Markdown 文本和压缩后的图片数组。
- **优势**：没有页眉页脚和错乱排版的干扰，大模型逻辑理解力暴涨；原图+上下文的形式比截屏整页 PDF 精准度更高。
- **锚点变更**：从 `[Page X]` 变为依赖 Markdown 的逻辑层级（如：指出问题所在的章节标题和确切原文片段）。

### 3.2 格式审查引擎 (Format Service) —— 实施“双轨制”
- **大模型轨道 (LLM)**：负责“语义与结构”审查。检查摘要是否缺失、中英文标题是否对应、图表编号逻辑是否连续等。
- **规则引擎轨道 (Rule Engine - 纯代码)**：负责“物理排版”审查。读取预先定义好的特定学校 JSON 格式规范（如中传：一级标题三号黑体、行距22磅），直接遍历 **Style AST** 节点进行严格代码比对。
- **合并输出**：将大模型找出的结构错误与代码找出的排版错误合并后返回。

### 3.3 AI 痕迹预警 (AITrace Service) —— 重构 Chunking 策略
- **输入变更**：由于放弃了 PDF 物理页码，切块策略必须改为 **按字数阈值 + 段落边界 (Paragraph)** 切分。
- **执行方式**：每约 2000 字切一刀，切口必须在段落结束处。每个 Chunk 的定位锚点变更为 `[章节名 - 第 N 段]`。输出结果时，精准定位到具体段落，而非模糊的页码。

### 3.4 参考文献核查 (Reference Service) —— 降维打击
- **提取重构**：**优先使用纯代码提取**。通过正则或 AST 遍历 DOCX，找到“参考文献”标题，直接薅出后面的纯文本列表。**匹配词需考虑兼容性**（如 `参考文献`, `References`, `Works Cited` 以及带空格的变体 `参 考 文 献`）。
- **Fallback 机制**：仅当学生排版极其混乱，代码无法命中“参考文献”标识时，才将文章末尾部分切出，交给低成本 Flash 模型（如 Gemini-Flash）进行兜底提取。
- **后续核查**：提取出的数组继续走现有的多源 API 瀑布流和裁判机制。

## 4. 存储与基础配置修改点
- **Supabase Storage**：修改 Bucket 允许上传的 MIME 类型，增加 DOCX。
- **Trigger.dev 环境**：需确保 Trigger 运行环境安装了 `sharp` 及其底层依赖（通常 Vercel/Node 环境自带兼容），并合理设置队列的 `concurrencyLimit` 防止 CPU/内存瞬时打满。
