# 参考文献真实性核查开发方案 v1.1

## 1. 概述 (Overview)

本方案在 v1.0 基础上进行了重大升级，引入 **OpenAlex** 作为核心数据源，形成了 **OpenAlex + ArXiv + CrossRef** 的 "三驾马车" 并行核查体系。
核心目标依然是识别论文中引用的虚假文献（Hallucinations），并在 UI 上以**批注（Annotation）**形式警告用户。

### 1.1 核心策略升级
1.  **三源验证 (Tri-Source Verification)**:
    *   **OpenAlex**: 主力数据源，覆盖面最广（聚合了 MAG、CrossRef、ArXiv 等），支持模糊搜索。
    *   **CrossRef**: 官方出版物权威源，用于校验正式发表的论文。
    *   **ArXiv**: 计算机/物理领域的预印本权威源，用于校验最新论文。
2.  **LLM 裁判 (LLM-as-a-Judge)**: 继续沿用 LLM 进行语义比对。
3.  **只报忧不报喜**: 仅对 **Not Found** 或 **Mismatch** 的文献生成批注。
4.  **英文优先**: 仅支持英文文献核查。

## 2. 前置准备 (Prerequisites)

### 2.1 OpenAlex API (新增)
OpenAlex 是目前最友好的免费学术图谱 API，无需注册 API Key，但建议在请求头中附带邮箱以进入 "Polite Pool"（获得更快的响应和更高的限额）。

*   **API Endpoint**: `https://api.openalex.org/works`
*   **配置**: 在请求 Header 中添加 `User-Agent` 或 `mailto` 参数。
    ```http
    User-Agent: ThesisAnno/1.0 (mailto:admin@thesis-anno.com)
    ```
*   **文档**: [OpenAlex API Documentation](https://docs.openalex.org/)

### 2.2 CrossRef API (Polite Pool)
*   **配置**: 同上，使用 `User-Agent` 标识身份。
*   **文档**: [CrossRef REST API Etiquette](https://github.com/CrossRef/rest-api-doc#etiquette)

### 2.3 ArXiv API
*   **限制**: 建议请求间隔 **3秒**。
*   **文档**: [ArXiv API User Manual](https://info.arxiv.org/help/api/user-manual.html)

## 3. 详细流程设计 (Workflow)

### 3.1 流程图 (Updated)
```mermaid
graph TD
    Start[PDF Buffer] --> Step1[LLM: 提取参考文献]
    Step1 --> List[文献列表 JSON]
    
    List --> Loop{遍历每条文献}
    Loop -->|中文| Skip[标记: Skipped (Non-English)]
    Loop -->|英文| Parallel[三源并行搜索]
    
    Parallel --> OpenAlex[OpenAlex Search]
    Parallel --> ArXiv[ArXiv Search]
    Parallel --> CrossRef[CrossRef Search]
    
    OpenAlex & ArXiv & CrossRef --> Merge[结果合并与择优]
    Merge --> Candidate[最佳候选结果]
    
    Candidate --> Step2[LLM: 智能比对 (Judge)]
    
    Step2 --> Decision{判定结果}
    Decision -->|Match| Pass[通过]
    Decision -->|Mismatch| Warning[生成警告批注]
    Decision -->|NotFound| Error[生成虚假警告]
    
    Warning & Error --> Result[加入最终结果集]
```

### 3.2 核心流程 (Two-Phase Workflow)
保持 v1.0 的两阶段设计，规避 Vercel 超时问题。

*   **Phase 1**: `POST /extract` (提取文献列表)
*   **Phase 2**: `POST /verify` (分批校验，10条/批)

### 3.3 Step 2: 三源并行搜索 (Tri-Source Search)

针对每一条待核查的文献，同时向三个数据源发起请求。

#### A. OpenAlex Search (主力)
*   **API**: `https://api.openalex.org/works`
*   **Query Strategy**: 使用 `search` 参数进行全文检索（OpenAlex 的搜索算法非常强大，类似 Google Scholar）。
    *   **Params**: `search="Title Fragment" FirstAuthorLastName`
    *   **Filter**: 可以尝试 `filter=title.search:xxx` 提高精确度，但建议直接用全局 search 提高召回率。
*   **优势**: 响应快，容错率高，元数据最全（包含 Citation Count, DOI, ArXiv ID）。

#### B. CrossRef Search (辅助)
*   **API**: `https://api.crossref.org/works`
*   **Query**: `query.title=...&query.author=...`
*   **优势**: 验证 DOI 的最终权威。

#### C. ArXiv Search (补充)
*   **API**: `http://export.arxiv.org/api/query`
*   **Query**: `ti:"..." AND au:"..."`
*   **优势**: 针对未发表的预印本。

#### D. 结果合并策略 (Merge Strategy)
为了给 LLM 提供最准确的判断依据，我们需要从三个源的结果中挑选一个**最佳候选 (Best Candidate)**。

1.  **Priority 1 (OpenAlex)**: 如果 OpenAlex 返回了高相关度结果（e.g. 标题高度匹配），优先采用。因为它聚合了 CrossRef 和 ArXiv 的 ID。
2.  **Priority 2 (CrossRef)**: 如果 OpenAlex 未命中，但 CrossRef 命中，采用 CrossRef。
3.  **Priority 3 (ArXiv)**: 如果前两者均未命中，但 ArXiv 命中，采用 ArXiv。
4.  **Fallback**: 如果三者都未找到，或相关度极低，标记为 `candidate: null`。

### 3.4 Step 3: LLM 智能比对 (Verification)

Prompt 逻辑微调，支持展示数据源来源。

```markdown
[原始引用]: {raw_text}
[检索结果]: {
   "source": "OpenAlex" | "CrossRef" | "ArXiv",
   "title": "...",
   "authors": [...],
   "year": "..."
}

判定逻辑:
1. status: "valid" - 确认为同一篇论文。
2. status: "mismatch" - 找到了相似论文，但关键信息（作者/年份）不符。
3. status: "not_found" - 检索结果为空或完全无关。
```

## 4. 开发计划 (Roadmap)

### Phase 1: API Clients 实现
1.  **`OpenAlexClient`**: 实现 `searchWorks(title)` 方法。
2.  **`ArxivClient`**: 保持原计划，注意限流。
3.  **`CrossrefClient`**: 保持原计划。
4.  **`SearchAggregator`**: 实现一个聚合层，并发调用上述三个 Client 并合并结果。

### Phase 2: Agent 逻辑
1.  集成 `SearchAggregator` 到 `ReferenceCheckAgent`。
2.  实现 Batch Verify 逻辑。

### Phase 3: 前端对接
1.  实现双轨 Hook (`useAiReview`)。
2.  UI 展示 Warning/Error 批注。

## 5. 风险控制

1.  **OpenAlex 匹配过宽**: OpenAlex 的搜索有时会返回相关但非目标的论文。
    *   *对策*: 依赖 LLM 进行最终把关 (Judge)，而不是盲信 API 结果。
2.  **网络并发压力**: 同时发 3 个请求 x 10 条文献 = 30 个并发请求。
    *   *对策*: 使用 `p-limit` 严格限制并发数为 5-10，避免瞬间流量过大导致 Vercel 报错或 API 封禁。
