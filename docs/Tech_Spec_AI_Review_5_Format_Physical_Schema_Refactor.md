# 格式审查物理轨：Schema 重构与边界划分方案

## 1. 背景与动机
在格式审查的双轨制架构中，“物理轨”承担着对纯物理排版（如字体、字号、行距等）的确定性校验。其核心逻辑是：**将用户的自然语言规范通过大模型抽取为结构化 JSON Schema（即“尺子”），然后与底层提取的 Style AST（即“被测物”）进行硬编码比对**。

为了应对各高校（如清华大学等）千奇百怪且极度复杂的格式要求，我们采用 **“最大化 Schema + 稀疏实例化”** 的设计策略。即：定义一个极其详尽的 Schema，大模型能抽出来的就填入，抽不出来的留空（`null`/`undefined`）。规则引擎在比对时，只对有值的字段进行严格校验，为空的直接跳过。

此方案旨在解决 Schema 表达能力不足的问题，同时明确划定“哪些能用物理轨查，哪些绝不能用物理轨查”的死线。

## 2. 核心 Schema 设计 (v2)
我们在 `lib/schemas/format-physical-extract.schema.ts` 中重构了 `FormatPhysicalExtract`。

### 2.1 基础能力基座 (PhysicalStyleSchema)
抽象出所有可复用的基础物理属性：
- **字体**：严格区分中文字体 (`font_zh`) 和西文字体 (`font_en`)。
- **字号与加粗**：`size_pt` (磅值), `bold` (布尔值)。
- **段落与间距**：`alignment` (对齐方式), `line_spacing_pt` / `line_spacing_multiple` (行距)。
- **段前段后**：`space_before_pt`, `space_after_pt`。
- **缩进**：`indent_first_line_chars` (字符数), `indent_first_line_pt` (磅值)。

### 2.2 作用域细分 (Contexts)
将基础能力分发到不同的上下文中，以便引擎在比对时知道“拿哪段规范去量哪段文本”：
1. `page_setup`: 页面设置（如从 `<w:sectPr><w:pgMar>` 提取的上下左右边距）。
2. `headings`: 1-6级标题规范（数组）。
3. `body`: 正文主体规范。
4. `caption`: 图表题注规范。
5. `references`: 参考文献列表正文规范。
6. `footnotes`: 脚注规范。

## 3. 边界划分与排雷清单（不可纳入 Schema 的规则）
为了防止规则引擎崩溃或产生大量误报，**必须严禁将以下两类规则强行塞入物理 Schema 中**。

### 3.1 纯语义/内容范畴（交由语义轨 LLM 处理）
这些要求关注“写了什么”，而非“长什么样”：
- **字数与数量限制**：如“题目控制在25个字以内”、“关键词3-5个”。
- **内容逻辑**：如“中英文摘要内容对应”、“图表编号分章连续（如图2.1）”。
- **特定标识符/格式写法**：如“参考文献写法统一不混用”、“用阿拉伯数字连续编排页码”。
- **章节内容要求**：如规定“引言”、“结论”必须包含的内容要点。

### 3.2 物理提取不可靠或极度复杂（物理轨暂时放弃）
受限于当前 OpenXML AST 提取的颗粒度，强行校验极易翻车：
- **封面与复杂绝对定位排版**：封面通常使用隐藏表格或文本框排版，段落 Context 极难被 AST 正确识别，强行应用规则会导致和正文混淆。
- **强制分页符校验**：如“中文摘要单设一页”。`<w:br w:type="page"/>` 位置随机，仅靠段落属性很难完美校验其逻辑正确性。
- **表格边框细节**：如“三线表的上下边线1.5pt，第三条线1pt”。需要深度解析和遍历 `<w:tblBorders>`，当前 AST 未涵盖。
- **高级编号逻辑**：如“不同页的脚注序号不连续”。
- **精细排版控制**：如孤行控制、允许西文在单词中间换行等。

## 4. 引擎升级核心：段内混排与局部异常治理 (Run-level Validation)

学生在撰写论文时，常因复制粘贴导致段落内出现局部字号突变或字体错乱（如一段 12pt 的正文里夹杂了几个 10.5pt 的字符）。为了精准捕捉这类“老鼠屎”，引擎的校验颗粒度必须从“段落级 (Paragraph)”下沉到“文本片段级 (Run)”。

### 4.1 被测物结构细化 (AST Run-span)
`DocxStyleAstNode` 必须完整保留段落内所有的 `<w:r>` (Run) 切片，不能仅提取首个 Run 或主导 Run 的样式。

```typescript
// 引擎内部处理的 AST 节点结构示例
type RunSpan = {
  text: string;
  font_zh?: string; // 对应 <w:rFonts w:eastAsia="..."/>
  font_en?: string; // 对应 <w:rFonts w:ascii="..."/>
  size_pt?: number;
  bold?: boolean;
};

type DocxStyleAstNode = {
  context: "body" | "headings" | "caption" | "references" | "footnotes";
  text: string; // 段落完整纯文本
  runs: RunSpan[]; // 核心：保留所有样式片段
  // ... 其他段落级属性（行距、缩进等）
};
```

### 4.2 引擎“逐字扫描”与启发式豁免逻辑
在应用 `body` 或 `caption` 规则时，引擎需执行两步走策略：

**第一步：段内一致性扫描（抓出复制粘贴残留）**
不急于用 JSON Schema 比对，先检查该段落自身的 `runs` 数组是否发生了无意义的突变。
- **计算主导样式 (Dominant Style)**：统计该段落中最长字符所使用的字号和字体，作为基准。
- **启发式豁免 (Heuristic Exemptions)**：
  1. **短前缀标签豁免**：如段首的 `"关键词："`、`"注："` 等短文本（≤8个字），允许其加粗或字号与主导样式不同。
  2. **中英文字体豁免**：Word 常将连续的英文/数字单独拆分为一个 Run。若该 Run 仅包含西文且符合 `font_en` 规范（或未违背整体视觉协调），不视为混排。
- **抛出异常**：若某个 Run 未被豁免，且其 `size_pt` 或 `font_zh` 偏离主导样式，直接生成严重错误（Error类型）：`“段内字体/字号不一致，疑似复制粘贴残留”`，并返回该 Run 的具体文本（`quote_text`）供用户定位。

**第二步：合规性比对（拿尺子量）**
在完成一致性检查后，提取该段落的“主导样式 (Dominant Style)”，与 JSON Schema 中对应的 `body` 或 `caption` 规则进行比对（包括字号、行距、段前段后等）。
比对时必须引入**魔法容差**：
- **磅值容差**：如要求段前 12pt，实际抽取为 12.05pt 时应视为合规（容差可设为 `±0.5`）。
- **字体别名映射**：如规则要求“宋体”，AST 抽取为“SimSun”时应判定匹配。

## 5. 下一步行动计划 (Next Steps)
本方案确立了“尺子”的形态，后续需推动底层“被测物”及“量法”的同步升级：
1. **升级 OpenXML 提取器 (Style AST)**：使其能够解析出段前段后间距、首行缩进、以及区分中英文字体（解析 `w:rFonts` 的 `w:eastAsia` 和 `w:ascii`）。**（已实现：`w:spacing` / `w:ind` + 样式链合并 → `DocxStyleAstNode` 的 `line_spacing_pt` / `line_spacing_multiple` / `space_*` / `indent_first_line_*`。）**
2. **升级物理规则引擎**：增加对新 Context（如参考文献、脚注）的识别与过滤逻辑；**并与 compile 输出的行距/缩进字段比对（容差见 `engine-baseline.json`）。**
3. **Prompt 调优**：更新大模型的抽取 Prompt，严禁其进行常识性推测（必须见文生义，无提及则置空）。
