## 一、目标与边界

| 项 | 说明 |
|----|------|
| 职责 | 统一管理 OpenRouter API Key、通过 `createOpenRouter` 懒加载单例、`getLLMModel(modelName)` 按配置动态选模型（不硬编码模型 ID）。 |
| 不在本层 | OpenRouter 429、全局并发与任务级重试（由 Trigger 外层编排，见 Spec §1、§2）。客户端未实现网络层自动重试；文件头注释已说明分工。 |
| 未包含 | `ReviewEngineError` 全量体系、`buildMessages`、`logic.service` 等真 LLM 调用（属后续 §3.2+）。 |

---

## 二、交付内容

### 1. 依赖

- `package.json` 增加：`@openrouter/ai-sdk-provider`（与现有 `ai@^6`、`zod@^4` peer 兼容）。

### 2. 核心代码

- **`lib/clients/openrouter.client.ts`**
  - `getOpenRouterProvider()`：懒加载单例，`compatibility: "strict"`，首次创建时校验 `OPENROUTER_API_KEY`。
  - `getLLMModel(modelName)`：trim 后非空校验；返回 `LanguageModel`（供后续 `generateObject` / `streamText`）。
  - 可选请求头：`HTTP-Referer` ← `OPENROUTER_HTTP_REFERER` 或 `NEXT_PUBLIC_APP_URL`；`X-Title` ← `OPENROUTER_APP_TITLE`（有值才设置）。

### 3. 环境与文档

- **`.env.example`**：补充 OpenRouter 区块（`OPENROUTER_API_KEY` 及可选 `OPENROUTER_HTTP_REFERER` / `OPENROUTER_APP_TITLE`）。
- **`README.md`**：环境变量示例中增加上述可选变量说明。

### 4. 本地校验（无网络请求）

- **`scripts/verify-openrouter-client.ts`**：断言空 `modelName`、缺失 Key、有 Key 时模型实例 `specificationVersion === 'v3'` 与 `modelId` 非空。
- **`package.json` 脚本**：`verify:openrouter`。

---

## 三、配置约定（后续接入时注意）

- `config/prompts.default.json` 中 `model_config.model` 若仍为短名（如 `gemini-1.5-pro`），需改为 OpenRouter 路由完整 ID（如 `google/gemini-2.5-pro-preview`），或在配置层做映射后再传入 `getLLMModel`。
