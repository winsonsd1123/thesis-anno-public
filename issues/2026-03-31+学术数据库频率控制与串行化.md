# 学术数据库频率控制与并发串行化

## 背景
系统运行在 Trigger.dev 多 Worker 环境：`orchestrateReview` 最多 5 个并发，`genericLlmBatchTask` 最多 3 个并发。
`verifyReferenceBatch` 内使用 `Promise.all` 并发调用所有文献的 `searchSourcesWaterfall`，最坏情况 3×10=30 个并发请求同时打向同一学术 API，远超 Semantic Scholar 0.33 RPS 等限额，导致 429 静默失败 → `candidate=null` → 文献误判。

## 修复内容

| 文件 | 改动 |
|------|------|
| `lib/integrations/academic/http.ts` | `fetchJson` 遇到 HTTP 429 时 sleep 2s 后重建 AbortController 重试一次，并打 `console.warn` 日志 |
| `lib/services/review/reference.service.ts` | `verifyReferenceBatch` 内学术 API 调用从 `Promise.all` 改为 `for...of` 串行，消除并发爆炸 |
| `lib/integrations/academic/arxiv.ts` | `max_results=1` → `5`，遍历多个 `<entry>` 做精确标题匹配（与其他源保持一致） |
| `lib/integrations/academic/pubmed.ts` | `retmax=1` → `5`，遍历多个 PMID 的 esummary 做精确标题匹配 |

## 并发控制现状（已有）
- `orchestrateReview`: `concurrencyLimit: 5`
- `genericLlmBatchTask`: `concurrencyLimit: 3`
- 参考文献批次：`for` 循环逐批串行（每次触发 1 个 `genericLlmBatchTask`）

## 效果
- 最坏情况：3 个并发 verify 任务各自串行查 10 条 → 某时刻最多 3 个并发学术 API 请求（可接受）
- 429 出现时有重试兜底，不会静默丢失
- 单批耗时略增（串行 vs 并发），但参考文献核查本身是分钟级任务，影响可忽略
