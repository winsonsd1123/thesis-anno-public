## 二、方案 1/3 全链路实现（数据 + Actions + UI + Realtime）

### 1. 数据库 / Supabase（MCP 迁移）

- `reviews` 表新增 **`domain text`**。
- 私有 Storage bucket **`thesis-pdfs`**（约 50MB、MIME `application/pdf`）。
- **`supabase_realtime`** 发布表 **`public.reviews`**（`replica identity` 已为 full）。
- 仓库内 SQL 备份：`docs/sql/20260321_reviews_workbench.sql`。
- **RLS 用户写库**：迁移已增加 `reviews` 对 `authenticated` 的 `INSERT` / `UPDATE` / `DELETE`（`auth.uid() = user_id`）；备份见 `docs/sql/reviews_rls_user_insert_update_delete.sql`。

### 2. 数据访问层

- `lib/dal/storage.dal.ts`：Service Role 上传/删除 Storage 对象，路径 `{userId}/{uuid}_{safeName}`，bucket **`thesis-pdfs`**。
- `lib/dal/review.dal.ts`：用户 JWT 路径统一 **`createClient()`** — 列表/详情/插入/重命名/删除/换 PDF/改领域/用户触发的「开始审阅」写库（`updateProcessingStart`），由 **RLS** 兜底防越权。
- `lib/dal/review.admin.dal.ts`：**仅**无用户会话的后台编排预留 — `createAdminClient()` 更新 `status` / `stages` / 完成 / 失败等；**禁止**在用户 Action 中误用。
- `lib/services/review.service.ts`：审阅领域唯一对外入口；Actions / RSC / Route Handler 只依赖本模块，由本模块分流 `reviewDAL` 与 `reviewAdminDAL`。

### 3. 业务层（Server Actions / Route）

- `lib/actions/review.action.ts`：`initializeReview`（页数由**浏览器** `pdfjs-dist` 解析后经表单字段 `pageCount` 提交，服务端校验范围）、`updateReviewDomain`、`renameReview`、`deleteReview`（删行后 `storageDAL.removeObject` 清理）、`replaceReviewPdf`（仅 `pending` 时换文件并更新元数据）。
- 审阅详情**只读查询**不走 Action：`GET /api/reviews/[id]` + `lib/client/fetchReviewRow.ts`（`Cache-Control: private, no-store`）。
- `lib/actions/trigger.action.ts`：`startReviewEngine` — 校验页数范围、估算点数、钱包余额；经 **`reviewService.updateProcessingStart`（用户客户端 + RLS）** 写 `processing` + 初始 `stages`；**扣点留 TODO**；若存在 `TRIGGER_SECRET_KEY` 则 `tasks.trigger("orchestrate-review", …)`（真编排见文档 2/3，本阶段为占位）。

### 4. 领域逻辑

- `lib/review/buildStaticPlan.ts`：静态计划 step id（无 LLM）。
- `lib/review/stagesUi.ts`：`stages` → 进度条与日志行 UI 数据。

### 5. 类型

- `lib/types/review.ts`：`ReviewRow`、`ReviewStatus`、`ReviewStageEntry`、`parseStages` 等。

### 6. 客户端

- `lib/store/useDashboardStore.ts`：`activeReview`、气泡链、`hydrateFromReview`、`patchFromServer`（Realtime 合并）。
- `lib/hooks/useReviewRealtime.ts`：`processing` 时订阅 `reviews` 的 `UPDATE`（消费 **`stages` 等列**，非单独的 `progress` 字段）。

### 7. 前端组件（`app/[locale]/dashboard/_components/`）

- `ReviewWorkbench`：真实历史、面板推导 + 手动 Tab、余额链、失败横幅与复制错误、接 `ReportViewer` / `ProgressConsole`。
- `HistorySidebar`：选中态、`+ 新审阅`、状态标签与色点；**重命名 / 删除**（下拉菜单 + 内联编辑）。
- `ReviewChatBoard`：上传、领域、计划、`startReviewEngine` 与错误文案映射。
- `ChatMessageRows.tsx`：对话区消息行渲染抽离（与 `ReviewChatBoard` 配合）。
- `UploadForm`、`DomainInfoBubble`、`PlanConfirmBubble`、`PaperCardBubble`、`ProgressConsole`、`ReportViewer` 等与数据流对齐。

### 8. 页面与工程配置

- `app/[locale]/dashboard/(withNav)/page.tsx`：拉取 `initialReviews` + 余额，传入 `ReviewWorkbench`。
- `next.config.ts`：`experimental.serverActions.bodySizeLimit: "52mb"`（与大 PDF 上传匹配）；**未**再使用 `pdf-parse` 的 `serverExternalPackages`。
- 依赖：浏览器解析使用 **`pdfjs-dist`**（无 `pdf-parse` / `@types/pdf-parse`）。

### 9. 国际化

- `messages/zh.json`、`messages/en.json` 下 `dashboard.review` 扩展（错误码 `errors.*`、历史状态、上传/保存中态、报告空块等）。

### 10. 质量

- `npm run build`、`npm run lint` 通过（`ReviewWorkbench` 内对 `setPanelOverride` 的 effect 使用 `eslint-disable-next-line` 说明「随审阅上下文重置手动 Tab」）。

## 三、与规格的差异说明（刻意对齐）

| 规格片段 | 实际做法 |
|---------|---------|
| 示例 `status: interviewing` | 使用库表枚举 **`pending`**（待点「开始审阅」） |
| Realtime 示例字段 `progress` | 使用 **`stages` JSONB** 驱动进度与日志 |
| 规格中 `@/utils/supabase/server` | 项目统一为 **`@/lib/supabase/server`** |
| 规格示例监听 `progress` 列 | 实际监听 **`stages`（及整行 UPDATE）** 驱动 `ProgressConsole` |
| 纯查询用 Server Action | 详情拉取改为 **`GET /api/reviews/[id]`** + `fetchReviewRow` |
