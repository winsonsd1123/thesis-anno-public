## 二、效果与问题（主观 + 客观）

### 1. 主观评价

- 端到端能跑通时，**输出质量尚未达到「可依赖的审稿意见」预期**（挑错深度、误报、与领域匹配度等需产品侧再标定与调 Prompt）。

### 2. 已观测客观问题

| 现象 | 说明 |
|------|------|
| OpenRouter **地区 / 模型可用性** | 部分 Gemini 模型返回 `403 This model is not available in your region`；与 **本地 dev 出口 IP**、**线上 OpenRouter 路由策略**有关，并非单纯「Trigger 区域」字段。多模态失败后走文本降级仍用同一 `model_config`，若模型本身不可用则仍会失败。 |
| 配置源 | 线上 Storage `prompts.json` 需包含 `logic_review_pass*`；**本地 default 与线上 Storage 可能不一致**，需 `init-config` 或后台同步。 |
| PDF 路径 | `pdf-parse` 文本降级对复杂排版论文信息损失大，**多模态成功时质量通常更好**，但依赖模型与地区是否支持。 |

### 3. 已做但未单独验收的项

- 未加自动化 E2E（真实 Key + 短样例 PDF）作为 CI 门禁；**效果**依赖人工与样例集。

## 三、建议后续优化（不承诺优先级）

1. **模型与地区**: 在 Admin 固定可替换的 OpenRouter 模型 ID；文档注明「若某模型报 region，请换模型或调整网络出口」。
2. **Prompt**: 按 `issue_type` 与 bad cases 做 A/B 与 Few-shot 约束，减少空话与重复。
3. **观测**: **已做** — `logic_result.observability`（条数、耗时）与 Trigger 可检索日志 `logic_review_pass_metrics`；若需计费/审计级落库可再接入 `llm_traces` 表。
4. **前端**: `ReportViewer` 仍为 JSON 展示；结构化展示与严重度筛选属需求文档已知的后续项。
