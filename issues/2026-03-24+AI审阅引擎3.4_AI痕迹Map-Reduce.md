## 一、与 §3.3（格式审查）的依赖结论

- **无业务先后依赖**：§3.3 为 `format.service.ts` 动态排版规则；§3.4 为分块 + `analyzeAiTraceChunk`，二者互不调用。
- **代码层面**：仅在 `ReviewAnalyzeContext` 上共享编排上下文类型；**不要求先完成 3.3 真实现** 才能交付 3.4。

## 二、已交付内容（实现层面）

| 模块 | 说明 |
|------|------|
| `lib/services/review/aitrace.service.ts` | `AiTraceResultSchema`（`issues`，含 `page`、`issue_type` 三枚举）；`analyzeAiTraceChunk`：`generateObject` + `getLLMModel`，`temperature` 上限 0.3；`mergeAiTraceChunkOutputs` 聚合各 chunk；`sortAiTraceIssues`：**先按 `page` 升序，同页再按 severity High→Medium→Low**。 |
| `lib/services/review/format.service.ts` | `AiTraceContextPayload` + `ReviewAnalyzeContext.aiTrace?`（`modelConfig`、`promptTemplate`）。 |
| `config/prompts.default.json` | `ai_trace_system`：`templates.zh` 对齐规格 §4.4（三维度 + `issues` 顶层 JSON 约定 + `--- [Page X] ---`）；`version` 1.0.1；`en` 同步为结构化输出说明。 |
| `trigger/utils/pdf-page-text.ts` | `extractPdfTextPerPage`：`pdfjs-dist` + 包内 `pdf.worker.mjs`；`buildAiTraceChunksWithPageAnchors`：默认 **每 5 页** 一块并注入页码锚点；`AITRACE_PAGES_PER_CHUNK` 可调整。 |
| `trigger/review-orchestrator.ts` | 加载 `ai_trace_system` 并注入 `ctx.aiTrace`（缺 key 则抛错）；**aitrace** 阶段：`extractPdfTextPerPage` → 分块 → `genericLlmBatchTask.batchTriggerAndWait`（`action: "aitrace_chunk"`，`dataBatch: [chunk]`）→ `mergeAiTraceChunkOutputs`；无页/无 chunk 时 `{ issues: [] }`。`format`/`logic` 仍为 `Promise.all`，aitrace 单独阶段。 |
| `genericLlmBatchTask` | 新增分支 `aitrace_chunk`：调用 `analyzeAiTraceChunk`；原有 `verify_references` 不变。 |

**验证**: `pnpm exec tsc --noEmit`、`pnpm run build` 已通过。

## 三、已知约束与后续

| 项 | 说明 |
|----|------|
| 线上 Prompt | `getPromptsDirect` 若从 Storage 拉取，需包含 **`ai_trace_system.templates.zh`**（及 `model_config`），否则编排会报错；与逻辑审查同理需 **init-config / 后台** 对齐。 |
| 端到端 | 真实 `aitrace_result` 需 **Trigger.dev + `OPENROUTER_API_KEY`** 跑 `orchestrate-review`；仓库内无样例 PDF，**未加** CI 级 E2E。 |
| 分块边界 | 相邻 chunk 边界处同一句话可能被拆到两块，理论上可能出现重复/漏检，属分块策略层后续优化项。 |
| 前端 | `ReportViewer` 仍为 JSON 展示；结构化列表与筛选属产品后续。 |
