# 安全审计修复

**日期**: 2026-03-16  
**来源**: [Security_Audit_Report_v1.0.md](../docs/Security_Audit_Report_v1.0.md)

## 已修复

### 1. Middleware Cookie 同步隐患（Critical）

- **位置**: `middleware.ts` L45-L52（修复前）
- **问题**: 先创建 Supabase 客户端（操作 response cookie 刷新 session），再创建 intlResponse，通过 `headers.forEach` + `headers.set` 复制 header。`headers.set` 会覆盖同名 header，Supabase 的多个 Set-Cookie（如 `sb-*-auth-token`、`sb-*-auth-token-code-verifier`）在复制时后者覆盖前者，导致 cookie 丢失。
- **后果**: 登录状态莫名丢失、Token 刷新失败、用户频繁掉线。
- **修复**: 采用 next-intl 官方推荐模式，先运行 `intlMiddleware(request)` 获取 `intlResponse`，再 `createClient(request, intlResponse)`，使 Supabase 的 cookie 操作直接作用于最终返回的 response，无需 header 复制。
- **验证**: curl 测试通过；未登录访问 `/zh/dashboard` 正确重定向至 `/zh/login`，Set-Cookie 头正常返回。

### 2. 支付 Webhook 支持 GET（Medium）— 无法按建议修复

- **位置**: `app/api/billing/webhook/zpay/route.ts`
- **问题**: Webhook 同时支持 GET，敏感参数（订单号、金额等）会出现在 URL 日志中，增加信息泄露、CSRF、重放风险。
- **审计建议**: 仅允许 POST。
- **实际情况**: Zpay 文档明确规定「支付结果通知」请求方法为 **GET**，平台仅以 GET 方式回调 notify_url，无 POST 选项。移除 GET 后回调失败，提示「需返回字符串 success」。
- **结论**: 保留 GET 支持，为 Zpay 平台限制，非我方可配置。已通过签名校验、金额校验、幂等处理缓解风险。

### 3. Host Header 注入（Medium）

- **位置**: `app/api/billing/create-order/route.ts`、`lib/actions/billing.actions.ts`
- **问题**: 信任 `x-forwarded-host` 构建回调 URL，代理配置不当时可被伪造，导致支付回调指向攻击者服务器。
- **修复**: 移除对请求头的依赖，统一使用环境变量 `NEXT_PUBLIC_SITE_URL` 构建 base URL。
- **验证**: 生产环境需在 `.env` 中正确配置 `NEXT_PUBLIC_SITE_URL`。
