# 格式物理轨：抽取、StyleAst、引擎迭代（2026-03-26）

本文档记录本轮对「NL→JSON 抽取 + OOXML styleAst + 物理规则引擎」的连续改进，便于后续查阅与回归。

## 1. 观测与落库

- **`analyzeFormat`**：`FormatReviewObservability` 增加 `physical_extract`（与 compile 输入一致的抽取 JSON）。
- **环境变量**：`FORMAT_PHYSICAL_EXTRACT_LOG=1` 时在服务端 `console.info` 打印格式化后的 `physical_extract`。
- **Supabase `reviews.format_physical_extract`**：此前迁移有列但无写入；在 `review.admin.dal` 的 `completeReview` / `setCompleted` 中同步写入，取值来自 `result` 内 `format_result.observability.physical_extract`（`lib/types/review.ts` 中 `pickFormatPhysicalExtractFromReviewResult`）。

## 2. 误报：正文规则套到封面/大标题

- **原因**：`body-default` 对所有非 heading 段落生效；封面行常为 Normal + 大字号，与正文 12pt 冲突。
- **基线** `config/format-rule-packs/engine-baseline.json`：`body_rule_skip_style_id_substrings`（Title/标题/题注等）、`body_rule_oversize_skip_delta_pt`（实际字号大于规则期望 + delta 时跳过 body）。
- **抽取 prompt**：明确 `body` 仅主体正文段落，不把封面/摘要/目录并入 `body`。
- **引擎**：`shouldSkipBodyParagraphRule` 在应用 body 规则前跳过上述情况。

## 3. 误报：标题被当正文（自定义样式名）

- **原因**：仅按 `Heading1` / `标题1` 等匹配段落样式；中文模板常用「章标题」「一级标题」等。
- **基线**：扩充各级的 `heading_style_patterns`（含中文与常见英文名）。
- **引擎**：`inferHeadingLevelByFormat` — 当样式未命中时，若**字号 + 字体**同时命中某条 heading 抽取规则，则视为该级标题，避免 body 误检。

## 4. 字号解析：`docDefaults`

- **原因**：`styles.xml` 中 `<w:docDefaults>` 未解析，导致默认字号/字体链不完整。
- **实现**：`parseStyles` 读取 `docDefaults.rPrDefault.rPr`，作为样式继承链底层，参与 `getStyleRPrResolved` / 无 run 段落。

## 5. 段内混排：Run 级 AST

- **问题**：整段只取第一个 run，无法反映「关键词：」黑体 14pt + 后续宋体 12pt。
- **类型**：`RunSpan`，`DocxStyleAstNode.runs`；段落级 `font/size_pt` 取 **dominant run**（文本最长）。
- **引擎**：body 且多 run 时 **逐 run** 检查字体/字号；短标签 run（≤8 字）不参与段内「字号不一致」启发式；段内多字号报「段内字号不一致」。
- **诊断**：`scripts/smoke-hybrid-docx.ts` 支持 `--dump-ast`，多 run 时打印各 run 明细。

## 6. 图表：抽取字段 + 段落上下文（方案 A）

- **抽取 schema**：`caption?: { font?, size_pt? }`（图注/表注统一一条，多数学校相同）。
- **prompt**：要求将图注/表注相关条款抽到 `caption`。
- **compile**：`caption-default` 规则，`match.kind === "caption"`。
- **StyleAst**：`ParagraphContext = "body" | "table_cell" | "caption"`。
  - 位于 `<w:tbl>` 内段落 → `table_cell`（**当前策略：跳过全部物理规则**，避免表内与正文混检）。
  - 题注样式名或文本前缀（图/表/Figure/Table 等）→ `caption`。
- **引擎**：`table_cell` 直接 continue；`caption` 只跑 `caption-default`；`body` 保持原 heading + body 逻辑。

## 9. 运维提示

- 线上若 prompts 来自 **Supabase Storage**，需同步 `format_physical_spec_extract` 的 `zh` 模板（含 `body` / `caption` 字段说明）。
- 历史已完成审阅行的 `format_physical_extract` 不会自动回填，需按需从 `result` 迁移或留空。

## 10. 已知局限（未在本轮解决）

- 表格内正文（`table_cell`）暂不检物理字号字体；若需校验表内体例，需单独规则与上下文设计。
- 图注/表注若未使用 Word 题注样式且标题不符合 `图N`/`表N` 等前缀启发式，可能仍落在 `body` 上下文。
